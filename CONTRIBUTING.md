## 行为准则

本项目倡导友善、尊重与建设性的协作氛围。

- 请保持礼貌、就事论事，避免人身攻击、歧视性言论、骚扰等行为。
- 如仓库包含 `CODE_OF_CONDUCT.md`，请以其为准；若没有，我们建议维护者补充并在本节链接说明。

## 在开始之前

为避免重复劳动与无效沟通，建议你在动手前先做以下检查：

1. **先搜索**：在 Issues 与 PR 中搜索是否已有相同/相近问题或变更。
2. **保持小步**：一次 PR 尽量只解决一个主题（一个 Bug / 一个功能点 / 一个重构点），便于审查与回滚。
3. **先讨论再大改**：若改动较大（例如影响 API/数据结构/公共接口），请先开 Issue 讨论并达成方案一致后再提交 PR。

## 提交 Issue（Bug/需求/问题）

> 沟通渠道：**仅使用 GitHub Issues/PR**（本仓库不提供其他即时通讯入口）。

### Bug 报告

请尽量包含以下信息（越完整越容易复现与修复）：

- **期望行为**：你认为应该发生什么？
- **实际行为**：实际发生了什么？（可附截图/日志）
- **复现步骤**：最小复现步骤（Step 1/2/3...）
- **环境信息**：
  - 操作系统与版本：
  - 运行环境/语言版本（如 Node/Go/Python 等）：
  - 浏览器与版本（如适用）：
- **影响范围**：是否阻塞你使用？是否有临时 workaround？

如果仓库配置了 Issue 模板（`.github/ISSUE_TEMPLATE/`），请优先按模板填写。

### 功能需求与改进建议

请描述清楚“为什么需要（Why）”与“想达到什么效果（What）”，并尽量补充：

- 使用场景/目标用户
- 期望的交互或输入输出示例
- 可选方案与取舍（如果你已经思考过）

### 提问与讨论

若是一般性使用问题，请尽量提供上下文：

- 你已经尝试过哪些方法？
- 相关配置/命令/日志片段
- 你希望得到的帮助类型（定位问题、给建议、评审方案等）

## 提交 Pull Request（PR）

### PR 提交流程

建议遵循以下步骤：

1. **Fork 仓库**并基于最新的默认分支拉取代码。
2. **创建分支**：使用有意义的分支名，例如：
   - `fix/xxx-bug`
   - `feat/xxx-feature`
   - `docs/xxx-doc`
3. **完成改动**：保持改动聚焦、避免无关格式化/顺手重构混入同一 PR。
4. **自检**：确保本地构建/测试通过（见下方[测试与自检清单](#测试与自检清单)）。
5. **提交 PR**：
   - 清晰描述变更动机（Why）与变更内容（What）
   - 关联相关 Issue（例如 `Closes #123`）
   - 提供必要的截图/日志/性能数据（如适用）

如果仓库配置了 PR 模板（`.github/PULL_REQUEST_TEMPLATE.md`），请按模板填写。

### 代码与文档规范

请在 PR 描述中说明你的改动是否涉及以下内容（如涉及请一并提供）：

- **对外接口变化**（API、CLI 参数、配置项、公开函数/类型等）
- **数据结构变化**（字段、序列化格式、兼容性）
- **文档更新**（README、示例、注释、变更记录）
- **破坏性变更**（Breaking Change）与迁移方式

### Commit 信息规范

建议使用 **Conventional Commits** 风格（便于自动生成变更日志与版本管理）：

- `feat: ...` 新功能
- `fix: ...` 修复问题
- `docs: ...` 文档变更
- `refactor: ...` 重构（非功能、非修复）
- `test: ...` 测试相关
- `chore: ...` 构建/工具/杂项

示例：

```
fix: 修复在空输入时的崩溃

原因：用户在未填写必填字段时点击提交会触发 panic
修改：增加参数校验并返回可读错误信息
```

### 测试与自检清单

提交 PR 前，请至少确认：

- [ ] 本地能成功构建/编译
- [ ] 相关测试用例通过（或在 PR 中说明缺失原因与替代验证方式）
- [ ] 代码风格/格式化符合仓库约定（如 lint/formatter）
- [ ] 变更包含必要的文档与注释（尤其是公共 API、复杂逻辑）
- [ ] 若改动影响行为/输出，已更新示例或 README

## 参与方式与维护者响应

- 维护者可能会要求你根据 review 意见进行修改；这属于正常协作流程。
- 若 PR 超过一段时间未响应，你可以在 PR 下礼貌地追加评论进行提醒。
- 若你的改动不适合合并（方向不一致、范围过大、缺少讨论等），维护者可能会建议拆分或先开 Issue 讨论。
