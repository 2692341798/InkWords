# Map-Reduce 分析架构设计规范

## 1. 背景与目标
针对超大型 Git 仓库，现有的单次全量文本拼接后交由大模型评估容易触发 DeepSeek 128k Token 截断，导致大模型无法获取项目全局结构。为了解决此问题并提升分析速度，我们引入“目录聚合拆分 + Goroutine池限流 + Map-Reduce层级汇总”的架构。

## 2. 核心架构设计

### 2.1 智能拆分 (Split)
- 改进 `GitFetcher`，不仅返回项目结构树 (Tree)，还会按目录和文件大小返回多个代码块 (Chunks)。
- **拆分策略**：基于目录层级进行聚合。单块内容最大字符数限制为 `300,000`。如果一个目录下的内容超限，则递归拆分为多个块；如果单个文件超限，则直接对其进行字符截断。
- 保证每个 Chunk 的独立性和完整性。

### 2.2 Map 阶段 (并发局部摘要)
- 对拆分后的每个 Chunk 进行并发的大模型分析，提取该目录/文件的核心功能、接口和数据结构。
- **并发控制**：使用 `golang.org/x/sync/semaphore` 限制最大并发 Goroutine 数量为 **3-5**。
- **容错机制**：如果某个局部任务因为 API 限流或网络异常失败，自动重试最多 **3** 次。如果重试仍失败，记录错误并跳过该部分，继续处理其他内容。
- **流式推送**：利用 SSE (`event: progress`) 细粒度推送每一个 Chunk 的开始、重试和完成状态，让前端实时感知分析进度。

### 2.3 Reduce 阶段 (全局梳理与大纲生成)
- 将所有 Map 阶段成功获取的局部摘要汇总为一份全局的上下文。
- 利用这份全局上下文，向大模型发送最终的 Reduce 请求，生成项目的全局架构设计大纲 (Series Outline)。

## 3. 交互与 API 变更
- `POST /api/v1/stream/analyze` 接口的底层逻辑将彻底重构，但对外暴露的 SSE 事件流更加丰富：
  - 增加 `chunk_analyzing`: 正在分析某一块。
  - 增加 `chunk_done`: 某一块分析完成。
  - 增加 `chunk_failed`: 某一块分析失败（含重试）。
- 前端在 `useBlogStream.ts` 中拦截并显示这些细粒度的进度信息，提升用户体验。

## 4. 依赖项
- 引入或继续使用 `golang.org/x/sync/semaphore`。
- 可选引入 `golang.org/x/time/rate` 如果需要令牌桶限频（目前可以通过 semaphore 结合简单重试来应对）。
