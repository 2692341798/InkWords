# 墨言博客助手 (InkWords) 产品需求文档 (PRD)

## 1. 产品概述

### 1.1 产品定位
“墨言博客助手 (InkWords)”是一款基于 DeepSeek API 的智能化技术博客写作辅助平台。它旨在通过强大的文档解析和 LLM 生成能力，将复杂的本地文档或 Git 仓库快速转化为高质量的技术博客。

### 1.2 核心价值主张
- **面向新手（小白友好）**：生成的文章结构清晰，对抽象概念强制补充代码解释，并配以无样式的 Mermaid 图表，确保读者能够一步步独立复现项目。
- **智能化拆解**：针对超过 5000 字容量的复杂项目（如官方教程、大型开源库），系统会自动评估并将其拆解为多篇系列博客，降低阅读门槛。
- **极致的安全与极简体验**：采用“阅后即焚”的安全策略（源文件解析后即删），并提供对标 Notion 体验的极简阅读与编辑环境。

## 2. 核心功能场景与用例

### 2.1 用户注册与登录流
- **场景描述**：用户首次访问或 Token 过期时进入系统。
- **业务流向**：为了降低注册门槛并保护 DeepSeek API 成本，系统**不支持游客生成博客，强制要求登录**。
  - **第三方一键登录**：支持通过 GitHub 或 微信 扫码授权一键登录/注册。
  - 登录成功后，前端保存 JWT 并重定向至工作台（Workspace）。

### 2.2 个人中心与用量管理
- **场景描述**：用户查看自己当月的生成额度、历史记录及第三方平台授权。
- **业务流向**：
  - **用量统计**：个人中心展示本月已使用的 Token 数或“剩余可生成博客篇数”。
  - **第三方绑定**：提供掘金、CSDN 等第三方内容平台的 OpenAPI 授权入口，绑定成功后方可使用“一键发布”功能。
  - **付费/升级入口**：提供明显的会员升级或充值入口，展示当前套餐等级。

### 2.3 小文件/单文档生成流程
- **场景描述**：用户上传小体积的 PDF、Word 或 Markdown 文档。
- **业务流向**：后端接收到文件后，提取纯文本并丢弃源文件（阅后即焚）。系统评估其复杂度较低，前端将携带文本内容请求后端的 SSE 流式生成接口，实时返回单篇结构完整、包含无样式 Mermaid 架构图的 Markdown 博客。

### 2.4 大项目/Git 仓库拆解流程
- **场景描述**：用户输入 GitHub/Gitee 仓库链接（如某个完整的开源项目或官方教程）。
- **业务流向**：
  1. **解析与过滤**：后端使用 `GitFetcher` 浅克隆仓库代码，过滤掉无关文件（如 `node_modules`, `.git`, 二进制文件），提取纯文本并在提取后立即销毁本地克隆（阅后即焚）。
  2. **评估与大纲规划**：系统评估其为复杂项目后，调用 `POST /api/v1/project/analyze`，LLM 首先生成一份“博客系列大纲”（如：基础篇、架构篇、实战篇）JSON。
  3. **并发生成**：用户在前端确认大纲后，系统通过 Goroutine 池并发生成各个系列博客，并通过 SSE 实时将各个篇章的生成进度推送给前端。

### 2.5 小白友好型文章生成流（知识点扩展）
- **场景描述**：在生成博客内容时，遇到枯燥、抽象的理论概念（如 React 的 Fiber 架构、Go 的 Goroutine）。
- **业务流向**：系统通过 Prompt 强制要求 LLM 自动补全解释性代码。生成的 Markdown 内容必须图文并茂，确保新手能独立复现文中的环境与代码。

### 2.6 异常中断与重试流
- **场景描述**：在生成过程中遇到网络断开或 DeepSeek API 超时。
- **业务流向**：系统必须立即捕获异常，并在前端通过 Toast 或状态栏提示用户。后端应自动保存已成功生成的段落或大纲到数据库中，提供“继续生成”或“重新生成”的断点恢复机制。

## 3. 非功能性与性能需求 (字数与速度指标)

### 3.1 内容长度限制
- **单篇字数控制**：为了保证“小白友好”的阅读体验并避免 Token 爆炸，单篇博客的最大长度限制为 **5000 字左右**。
- **强制拆分机制**：当系统评估到源文档或拉取的 Git 代码如果转换为博客会超过 5000 字时，必须**强制**将其拆分为几篇小于 5000 字的系列文章，生成系列博客。

### 3.2 响应与处理速度 (TTFB)
- **文档解析与首字节响应**：用户点击“生成博客”后，后端需在 **10 秒内**完成文档/代码库的文本提取与过滤，并开始输出 Markdown 流（首字节到达前端）。

### 3.3 输入体积控制
- **本地文件限制**：为了保障内存解析性能，拖拽上传的本地文档（PDF, Word）单文件体积限制为 **50MB**，Markdown 等纯文本限制为 5MB。
- **代码仓库限制**：Git 仓库克隆前需通过 OpenAPI 获取其大小或在克隆时设置深度，过滤掉超大型二进制文件，确保解析的纯文本源码总量控制在合理的上下文窗口内。

### 3.4 渲染性能控制
- **Mermaid 复杂度限制**：通过 Prompt 约束 LLM，生成的架构图和流程图的节点数量建议控制在 20-30 个以内。若逻辑过于复杂，需拆分为多张小图，防止前端由于渲染超大图表而卡顿。

## 4. 历史管理与编辑逻辑

### 4.1 基本记录与查看
- **历史归档**：所有通过系统成功生成的博客，都将在 PostgreSQL 数据库中自动保存一份快照。
- **列表导航**：用户在左侧边栏（Sidebar）可查看过往生成的文章列表（按时间倒序排列）。对于被拆分的大项目系列文章，列表中需提供“折叠/展开”的分组展示逻辑。

### 4.2 二次编辑与增量修改
- **Markdown 编辑器**：系统提供类似 Notion 体验的所见即所得/双栏 Markdown 编辑器。用户在预览生成结果后，可直接对不满意的段落、代码块或 Mermaid 图表进行手动修改。
- **保存与覆盖**：用户手动修改的内容支持“自动保存”和“手动覆盖（Ctrl+S）”，并同步更新到数据库中。

### 4.3 导出与下载功能
- **导出格式**：支持将单篇或整个系列的博客，一键导出为标准的 `.md` 文件或排版整洁的 `.pdf` 文件。
- **图表兼容**：在导出时，系统需确保 Mermaid 图表代码块保持纯净格式，或者直接将其渲染为图片（针对 PDF 导出），以保证跨平台阅读的兼容性。

## 5. 异常处理与失败场景机制

### 5.1 解析阶段失败
- **文件格式错误/损坏**：若用户上传了加密的 PDF 或不支持的文件类型，后端立即返回 400 错误，前端弹窗提示“文件解析失败，请检查格式或解除密码”。
- **Git 仓库拉取失败**：若仓库为私有或 URL 错误，提示“仓库拉取失败，请检查权限或地址是否正确”。

### 5.2 生成阶段失败 (LLM 异常)
- **API 超时或欠费**：若调用 DeepSeek API 遇到超时、频率限制（Rate Limit）或 Key 无效，前端需明确提示“AI 生成接口暂时不可用，已为您保存当前进度”。
- **内容违规拦截**：若提取的代码触发了 LLM 的安全审查，中止生成，提示用户“文档内容包含敏感信息，无法生成”。

### 5.3 断网与数据保护
- **SSE 流式中断**：如果前端在接收流式数据时断网，系统需提供“恢复生成”按钮。后端需保证之前已生成的内容成功落库，避免用户从头开始等待。
