## 1. 产品定位
墨言知识训练平台 (InkWords Trainer) 是一款面向个人知识沉淀与复习的本地知识工作台。它能够将本地文档（PDF/DOCX/Markdown/TXT/ZIP 课件包）或开源 Git 仓库（代码/官方教程）整理为可持续增长的 Obsidian 知识库，并通过“知识漫游复习”形成训练闭环；同时保留将内容进一步生成小白友好、可复现的系列技术博客作为可选输出能力。

## 2. 核心功能需求

### 2.1 文档解析与生成
- **本地文件解析**：用户可上传 PDF/DOCX/Markdown/TXT 格式的技术文档，系统提取文本内容，单文件最大支持 888MB，并在前端与网关层进行统一拦截校验。针对大于 5 万字符的长文档，系统将自动进行分块（Map-Reduce）分析。
- **ZIP 课件包解析**：用户可上传单个 ZIP 课件包；系统会自动扫描压缩包中的受支持文档与代码文本文件，完成白名单筛选、内容去重、按顺序聚合，并输出解析摘要（扫描数、保留数、去重数、忽略数、失败数）后继续进入大纲分析。
- **Git 仓库解析与模块选择**：用户输入公开的 Git 仓库地址，系统极速拉取目录结构并智能扫描一级核心代码目录，结合仓库 `README` 调用轻量级 AI 智能推断每个目录的作用并生成中文简介，以卡片形式展示。用户可按需勾选模块进行定向解析，加速处理并避免内容超限。
- **大纲拆解与规模评估**：系统根据解析到的长文本（选中的 Git 模块或超大本地文件），智能评估内容规模，并**强制拆分为系列博客大纲**（包含章节标题、摘要、排序等）。
- **内容生成与系列串联**：
  - 基于生成的大纲，系统以**并发**的方式调用 DeepSeek API 为每个章节生成具体的独立博客内容。
  - 支持“创作场景”选择：`电子书解读 / 开卷复习 / 小白教程`，用于表达本次任务目标；并继续支持“文章类型/风格”选择（如通用技术博客/小白手把手/备考复习）作为呈现风格层。
  - 对于文件上传来源，系统必须在 Analyze 阶段自动识别内容类型，并锁定匹配的动态提示词 profile；例如沟通/心理学书籍优先走“心理学经典解读”，考试材料优先走“开卷复习材料”，技术手册优先走“技术资料讲解”。
  - 所有独立博客生成完毕后，系统将自动触发一次 AI 调用，生成一篇全局的“导读/总结”文章，将单篇博客串联为完整的专栏系列。
  - **单点聚焦与深度重写**：强制单篇博客仅深入剖析一个核心技术点。对于源码更新后的已有章节，系统将启动**松散参考重写 (Regeneration)**，提取旧版博客注入大模型，基于最新源码保留优质解释并更新代码逻辑。
  - 生成的内容必须包含代码级剖析（逐行解释）和可复现的步骤。
  - 对于抽象的理论概念，必须提供对应的代码示例或生活化比喻（小白友好）。
  - 支持 Mermaid 图表（强制要求无自定义样式）。
- **实时进度展示**：前端需提供基于 Server-Sent Events (SSE) 的多卡片流式进度展示，支持用户实时预览多个并发章节的生成状态和打字机效果。
- **提示词类型可见性**：当文件 Analyze 完成后，前端必须在大纲区展示当前已锁定的“提示词类型”中文标签与分类原因，让用户知道系统为什么把当前资料当作某一类内容来解释。
- **生成器步骤约束（2026-05-29 更新）**：
  - 顶部流程固定三步：`选择来源 -> 配置解析 -> 确认大纲`。
  - 解析/分析期间仍归属“配置解析”步骤，不得切换到独立“处理进度”页面。
  - 写作生成期间仍归属“确认大纲”步骤，不得切换到独立“处理进度”页面。
  - 上传 ZIP/文档后必须先停留在“配置解析”让用户选择创作场景，再由用户显式点击“生成大纲”触发分析。
- **正文纯净输出**：所有 AI 生成正文必须对用户可见内容做净化，严禁将模型思考标签、推理过程或对话式前言混入最终 Markdown 正文与保存结果。

### 2.1.1 创作场景需求（`scenario_mode`）
- **电子书解读**：适合长文档、经典著作、概念材料。输出应更偏向篇章结构、观点提炼、白话解释与现实映射。
- **开卷复习**：适合课件、实验指导、考试范围资料。输出应更偏向考点清单、步骤模板、易错点和速查结构。
- **小白教程**：适合 Git 仓库、项目教程、官方文档。输出应更偏向环境准备、目录结构、主链路、排错说明和可复现步骤。
- **默认策略**：
  - Git 仓库默认推荐“小白教程”。
  - 文件上传默认推荐“电子书解读”。
  - 用户可以在生成器页面显式手动切换，不应被系统默认值锁死。
- **交互约束**：
  - 用户只能在生成大纲（Analyze）之前手动切换 `scenario_mode`。
  - Analyze 返回大纲后，前端必须隐藏场景选择器，并在大纲区展示当前场景的只读中文标签。
  - 大纲生成后的正文生成必须沿用同一次 Analyze 已锁定的 `scenario_mode`，不允许中途修改。
- **兼容要求**：
  - 旧客户端不传 `scenario_mode` 时，后端必须自动兜底并保持链路可用。
  - `scenario_mode` 必须同时影响 Analyze 与 Generate，避免大纲和正文风格割裂。

### 2.1.2 动态提示词类型需求（`prompt_profile`）
- **目标**：在“创作场景”之外，系统还必须根据文件内容本身识别更细粒度的内容类型，用于决定系统角色、解读重点与正文组织方式。
- **适用范围**：
  - 仅对文件上传来源强制启用（PDF/DOCX/Markdown/TXT/ZIP 聚合文本）。
  - Git 仓库链路当前仍以 `scenario_mode + article_style` 为主，不要求先做内容类型分类。
- **最小能力要求**：
  - Analyze 前或 Analyze 过程中，系统必须对文件内容做一次轻量分类。
  - Analyze 结果必须返回 `prompt_profile_key`、中文展示名、`document_kind` 与分类原因。
  - Generate（单篇、系列章节、系列导读）必须沿用同一次 Analyze 已锁定的 profile，不能在生成时重新漂移到其它提示词类型。
- **回退要求**：
  - 当分类器不可用、内容为空、返回非法值或解析失败时，系统必须按 `scenario_mode` 回退到默认 profile，并显式告诉用户当前为默认提示词。
- **前端交互约束**：
  - 大纲出现后，用户可以看到“当前创作场景”和“当前提示词类型”两个只读标签。
  - 当前版本不要求用户手动切换 prompt profile，默认由系统自动识别并锁定，以减少对小白用户的额外认知负担。

### 2.2 博客管理
- **历史记录列表**：用户可查看所有已生成的博客（按系列父节点和子章节组织）；当系列章节生成中断或局部失败时，系统仍需保留系列父子结构，避免历史区退化为仅显示导读。
- **流程型工作入口**：当用户未打开具体博客编辑器时，系统默认先进入引导式 `HomeEntry`，而不是直接把用户扔到生成器表单；首页需要能在“生成博客 / 知识复习”两类主路径之间切换，并保留继续任务与最近记录。
- **Markdown 编辑器**：内置 Markdown 编辑器，支持用户对生成的博客进行二次编辑与实时预览，并在输入停止后自动保存到数据库。
- **语音输入**：在写博客编辑器中支持语音转写输入，转写内容实时写入正文，提升手写写作效率。
- **润色**：支持对当前草稿进行全文润色，前端展示“润色预览”，并支持一键将润色结果应用到正文。
- **手写博客入口**：支持一键创建空白草稿并进入编辑器，作为手写写作的入口。
- **导出与下载**：支持将单篇博客或整个系列的博客打包下载为 Markdown 格式（ZIP 压缩包）。
- **直通 Obsidian 知识库**：支持一键将生成的单篇博客（附带标准 YAML Frontmatter）直接导出至本地 Docker 挂载的 Obsidian Vault 目录，实现个人知识库的无缝沉淀。
- **系列批量 Ingest（知识复利）**：支持一键将整个系列批量导入 Obsidian，并遵循 Karpathy LLM Wiki Pattern：系列作为 `sources/`，章节作为 `concepts/`，并自动抽取关键 `entities/` 与 `concepts/` 建立双向链接网络，同时更新 `wiki/index.md`、`wiki/log.md`、`wiki/hot.md`。

### 2.3 知识漫游复习
- **独立入口**：登录后，侧边栏必须提供“知识漫游复习”入口，进入独立主视图，不与博客生成页混在同一工作流中。
- **单步聚焦**：无论是知识复习还是博客生成，页面主区同一时间只展示当前步骤；其他步骤通过统一步骤条做预览或进度反馈，避免多个主任务区域同时平铺。
- **两种开始方式**：
  - 随机抽题：用户手动随机抽取一篇可复习文章，且“再抽一篇”在存在其它候选时应尽量换到不同文章。
  - 手动选文：用户可检索并选择知识库中的候选笔记开始复习。
- **两种训练模式**：
  - `light_recall`：更偏“你先讲，我再给提示”的轻提示复述。
  - `detailed_qa`：更偏逐轮追问与细致问答。
- **会话闭环**：
  - 创建 session 后，页面必须展示开场问题、当前文章的结构化复习快照、本轮目标、可请求提示、可提交回答、可主动结束训练。
  - `detailed_qa` 的追问必须基于当前文章内容推进，至少覆盖“主线问题 -> 关键概念/步骤 -> 场景/迁移解释”三类方向，而不是固定模板问答。
  - 用户每次提交回答后，系统必须明确返回“本轮判断、答到的点、漏掉的点、下一步建议”，让用户知道自己到底说对了多少。
  - 所有阶段反馈、提示、总结文案都必须使用中文，避免暴露底层模型的英文或调试信息。
  - 最近复习记录必须能看到刚完成的 session 摘要，便于用户复盘。
- **数据边界**：
  - 复习正文来源于 Obsidian `wiki/` 下的知识页面。
  - 业务状态、反馈结果与轮次记录持久化到 PostgreSQL，不直接把整篇原始笔记全文复制进数据库。

### 2.4 用户与授权
- **第三方登录**：支持通过 GitHub OAuth 进行授权登录。
- **普通注册/登录**：支持基于邮箱/密码的常规账号注册与登录。
- **安全防爆破机制**：注册和登录强制校验本地图形验证码防刷；连续 5 次密码错误将自动锁定账号 15 分钟，并在密码输入时提供实时强度校验提示。
- **Token 鉴权**：系统采用 JWT 机制进行用户身份的验证和路由拦截，支持“保持登录状态”签发长效 Token。
- **用户个人中心与仪表盘**：新增仪表盘展示用户消费的 Tokens 数量、预估费用（1元/1M Tokens）、文章总数、总字数，并以图表直观展示用户使用最高频的技术栈排行。允许用户修改用户名及上传头像。

## 3. 非功能性需求
- **并发与限流保护**：并发生成博客时，使用环境变量动态控制并发数（默认 10），最大上限可调至 50，以充分利用大模型的性能并防止 API 限流（429 错误）和服务器内存溢出。
- **Token 长度保护**：拥抱 DeepSeek V4 的 1M Token 上下文能力，向大模型发送的源内容字符数限制提升至 2,000,000 字符以内，以保证超大型项目的完整解析能力。
- **隐私与安全**：用户上传的文件仅在内存中处理，解析完成后必须立即销毁，绝不持久化存储物理文件实体。
- **容器化部署**：前端、后端和数据库必须通过 Docker Compose 一键启动（推荐 `docker compose --env-file backend/.env up -d --build`），确保本地与线上环境的完全一致性。前端由 Nginx 提供服务并代理后端 API，默认仅暴露 `http://localhost` 作为公开访问入口。
