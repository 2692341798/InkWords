# GitHub 仓库解析重构与系列文章生成方案设计 (Design Spec)

## 1. 背景与目标
原有的 GitHub 仓库解析功能存在“解析子目录功能鸡肋”、“不知有哪些目录可供解析”的痛点。为了解决这一问题，本项目将原本“一步到底”的解析流程拆分为两步。
第一步：拉取仓库代码，智能提取核心代码目录，并使用轻量级 AI 总结目录简介，以卡片选项形式展示给用户。
第二步：用户勾选所需模块后，后端再根据所选内容进行分析、生成大纲、撰写多篇独立博客，最后生成一篇导读文章将它们串联，形成系列文章。

## 2. 架构与 API 设计
将原有的一键解析流程拆分，新增预扫描接口，改造解析接口。

### 2.1 预扫描接口 (Pre-scan)
- **Endpoint**: `POST /api/v1/project/scan` (或复用现有路由路径如 `/api/v1/github/scan`)
- **Input**: `{"github_url": "https://github.com/xxx/yyy"}`
- **Process**:
  1. 后端调用 `git clone` 将代码拉取到本地临时目录。
  2. 扫描项目根目录，智能过滤非业务代码目录（如 `.git`, `docs`, `assets`, `node_modules` 等），保留核心目录（如 `src`, `internal`, `pkg`, `cmd`, `lib` 等）。
  3. 并发调用轻量级 AI（如 DeepSeek-Chat），为每个提取出的目录生成一句简短的功能简介（不超过 30 字）。
  4. 将目录名、路径、简介组合成卡片数据结构，返回给前端。
- **Output**: 
  ```json
  {
    "code": 0,
    "data": {
      "repo_url": "https://github.com/xxx/yyy",
      "modules": [
        {"path": "src/api", "name": "api", "description": "处理所有对外的 HTTP 请求及路由定义。"},
        {"path": "src/services", "name": "services", "description": "核心业务逻辑层，包含用户、订单等服务。"}
      ]
    }
  }
  ```

### 2.2 解析与生成接口 (Analyze & Generate)
- **Endpoint**: `POST /api/v1/project/analyze` (流式 SSE 接口)
- **Input**: 
  ```json
  {
    "github_url": "https://github.com/xxx/yyy",
    "selected_modules": ["src/api", "src/services"]
  }
  ```
- **Process**:
  1. 后端接收用户选中的模块列表，针对这些特定的目录路径提取代码文本。
  2. 针对每个选中模块，分别执行：分析架构 -> 生成大纲 -> 撰写单篇博客。
  3. **系列文章串联**：当所有独立模块博客生成完毕后，基于它们的总结信息，最后再触发一次 AI 调用，生成一篇全局的“导读/总结”文章，该文章内部会通过链接关联上述单篇博客。
- **Output**: SSE 流式推送，分别推送每个模块的生成进度，最后推送导读文章的生成状态。

## 3. 数据库与存储 (Storage)
由于多篇博客需要串联为“系列”，数据库 `blogs` 表需要进行逻辑上的层级划分。
- 可在 `blogs` 表中新增一个可选字段 `parent_id` (或通过 `series_id` 关联)，用于标识该文章属于哪个系列。
- **导读文章** 作为 Parent (或不带 `parent_id` 但作为系列的入口)。
- **模块文章** 作为 Children，其 `parent_id` 指向导读文章的 ID。

## 4. 前端交互设计 (Frontend)
- **第一步：输入与扫描**
  - 页面提供一个 GitHub URL 输入框。
  - 用户点击“分析仓库”，前端调用 `/scan` 接口，进入 Loading 状态。
- **第二步：模块选择卡片**
  - 接口返回后，页面展示“模块选择”视图，以卡片网格的形式展示项目目录。
  - 每个卡片显示：目录名 (`name`)、AI 简介 (`description`)。
  - 用户可点击勾选多个卡片。
- **第三步：生成系列博客**
  - 勾选完成后，点击“生成系列博客”按钮，调用 `/analyze` 接口（SSE），进入多步骤进度条页面。
  - 进度条需支持展示多篇文章的并行/串行生成状态（如：“正在生成 api 模块博客”、“正在生成 services 模块博客”、“正在生成系列导读文章”）。

## 5. 风险与缓解策略
- **仓库克隆耗时过长**：针对超大型仓库，只拉取默认分支且 `--depth=1`，若克隆失败则返回友好错误。
- **并发调用 AI 被限流**：生成目录简介时，目录数量可能较多，需使用 Goroutine 池与 semaphore 信号量限制并发数（如最大并发 5）。
- **临时文件清理**：确保 `/scan` 接口和 `/analyze` 接口执行完毕或发生错误后，均有 `defer os.RemoveAll(tmpDir)` 逻辑，防止磁盘被撑爆。

## 6. 后续计划
1. 在后端实现 `Scan` 接口与目录智能过滤算法。
2. 改造前端页面，增加模块选择卡片交互。
3. 改造后端解析接口，支持接收 `selected_modules`，并串联生成导读文章。
4. 更新数据库 Schema，加入系列文章关联字段。
