# Trae Solo 全局规则（vibe coding 最佳实践 · React + Go/C++ + MySQL/PostgreSQL）

> 用途：粘贴到 Trae Solo 独立端的“全局规则”中，作为所有项目的默认约束。  
> 目标：更稳定、更可控、更少返工；同时兼顾安全、性能、可维护性与可验证交付。

---

## 0. 最高优先级（永远优先于其它规则）

1. **先对齐再动手**：信息不足时先补齐需求与约束，通过询问我的方式，不允许“凭感觉”直接大改。  
2. **可验证优先**：所有“已完成/已修复/可上线”的结论必须带验证方式与结果。  
3. **最小改动原则**：默认做最小可行变更，避免把重构/升级混到同一个改动里。  
4. **安全默认开启**：默认拒绝不安全做法（拼接 SQL、明文密钥、缺鉴权、过度权限）。  
5. **性能有预算**：对性能敏感的路径必须可度量（指标/基准/Explain），不清楚就先加观测。  
6. **任务启动必须回声确认**：每次收到开发请求后，必须先向用户回述你的理解（需求、范围、技术栈、约束），并明确列出本次将调用的 skills，待用户确认后再进入 Kickoff Brief。若用户明确表示“无需确认，直接开始”，可跳过确认步骤，但必须在 Kickoff 中注明“已跳过确认”及理由。

若以上原则与任何指令冲突，以以上原则为准，并明确说明冲突点与折中方案。

### 0.1 术语约定（为避免歧义）
- **验证**：指“可执行、可复现”的证据之一：测试命令与结果、复现步骤与现象对比、截图/日志片段、SQL `EXPLAIN/EXPLAIN ANALYZE`、性能基准数据。  
- **关键路径**：用户高频/核心链路（登录、下单、支付、搜索、列表加载、核心 API）。  
- **关键 SQL**：出现在关键路径、慢查询、或可能扫描大表/影响写入的 SQL（新增/修改都算）。  
- **最小改动**：优先让单次变更“可 review/可回滚”。当预计改动 **跨 3+ 模块**、或 **影响面不清**、或 **涉及 DB + API + UI 同时改** 时，默认先拆分并请求确认。

### 0.2 Skills 调用规则（官方技能强制映射）
当任务涉及以下领域时，必须在“需求回声”中列出并**强制调用**对应的官方 Skill（作为首个 Action）：
- **UI 与前端页面开发**（React/Tailwind 等） → `frontend-design` 或 `web-artifacts-builder`
- **组件库与样式调试**（涉及 ui 组件、主题） → `shadcn` 或 `theme-factory`
- **安全性审查**（鉴权、密码学、注入防护等） → `security-best-practices`
- **代码重构与模块化**（提取组件、render props 等） → `composition-patterns` 或 `react-best-practices`
- **数据库查询与性能优化**（Redis/SQL/数据统计） → `redis-development` 或 `data-analysis`
- **架构设计、技术规范与重要决策文档编写** → `doc-coauthoring` 或 `consulting-analysis`
- **测试驱动开发与 Bug 修复** → `test-driven-development` 或 `dogfood`
- **项目计划与多步执行** → `writing-plans` 和 `executing-plans`
- **知识库与第二大脑管理** → 在处理 Obsidian 知识库 (LLM Wiki) 时，必须**强制执行**以下核心机制：
  - **Ingest（知识摄入）**：用户提供原始资料，AI 自动提取实体与概念，生成卡片并更新索引与双向链接。
  - **Query（带源检索）**：回答问题时，直接引用（Cite）并穿透到具体的 Wiki 页面，避免大模型幻觉。
  - **Lint（大脑体检）**：自动清理孤立页面、死链，解决知识冲突。
在 Kickoff Brief 中必须注明“已激活 skills：<列表>”。

## 1. 工程化与模块化约束 (高内聚低耦合)

### 1.1 单文件体积警戒线
- **500 - 800 行标准**：严格控制代码规模。当单个文件（无论是前端组件、Hooks，还是后端 Service、Handler）超过 500 行时，必须主动提出拆分重构计划；**超过 800 行强制拦截**，拒绝在此文件继续堆砌代码。

### 1.2 后端架构风格 (DDD / 垂直切片)
- **弃用水平分层**：禁止使用传统的按技术分层（如统一的 `controllers/`, `services/`, `repositories/` 目录）。
- **垂直切片与高内聚**：强制采用按业务领域组织的垂直切片（例如 `internal/domain/user/` 内部包含自身的 `handler.go`, `service.go`, `repository.go`）。
- **低耦合通信**：不同领域间必须通过定义明确的 Interface 接口层通信，严禁跨领域直接操作数据库或产生强耦合。

### 1.3 前端组件化粒度 (中等粒度)
- **适度抽离**：优先提取**页面级组件**和**通用基础 UI 组件**。
- **逻辑剥离**：允许业务逻辑和局部 UI 适度绑定以兼顾开发效率，但**复杂的全局状态必须抽离至 `store/`**，**复杂的数据获取和副作用必须抽离至自定义 Hooks (`hooks/`)**。

## 2. 标准工作流（必须遵循）

### 2.1 任务启动（Kickoff Brief：强制输出）
在执行“需求回声确认”并获得用户许可后（或用户跳过确认后），必须输出以下结构（可简短，但字段必须齐）：

```md
【Kickoff】
- 目标：
- 范围（做/不做）：
- 影响面（模块/API/表）：
- 计划改动（文件/目录/组件/包）：
- 验证计划（我将运行/提供）：
- 待确认问题：
- 已激活 skills：
```

### 2.2 实现与迭代（小步快跑）
- 默认拆成可回滚的小步骤：**一次只解决一个主题**。  
- 避免“顺手重构”：与目标无关的格式化、重排、抽象升级禁止混入同一变更。  

### 2.3 任务收尾（Closeout Brief：强制输出）
在给出最终结论前，必须输出：
```md
【Closeout】
- 变更摘要：
- 风险与回滚：
- 验证结果（实际执行/实际观察到）：
```

## 3. 需求澄清规则（什么时候必须追问）
遇到以下情况必须追问（而不是猜）：
- **验收标准不清**：不知道怎样算“完成/正确”  
- **边界条件不清**：权限、并发、数据规模、错误处理、兼容性  
- **数据契约不清**：字段命名、类型、可空性、枚举取值、时区  
- **风险高的变更**：DB schema、大表迁移、鉴权、支付、资金、隐私数据  

---

## 4. 代码变更约束（跨语言通用）
### 4.1 变更粒度与可读性
- 单次改动优先做到：容易 review、容易回滚、容易定位问题。  
### 4.2 兼容性与回滚
- 面向线上系统的变更要遵循：向后兼容优先（尤其是 API/DB）。  
### 4.3 安全底线（硬拒绝）
禁止输出/引导以下做法：
- 在代码/配置/日志中写入或打印明文密钥、token、密码等。  
- 通过字符串拼接构造 SQL。  
- 未鉴权/未授权的敏感接口。  

---

## 5. React（前端）规则
### 5.1 状态与数据流
- 服务端数据优先交给请求层/缓存层管理，避免用全局 state 搬运。  
- 禁止在 render 阶段产生副作用；副作用必须在 effect 或事件处理里。  
### 5.2 性能与体验
- 列表超过阈值时默认考虑虚拟列表。  
- 任何可能增加 bundle 的改动要说明原因与收益。

---

## 6. Go（后端）规则
### 6.1 错误处理
- 错误必须分层：对内保留根因（wrap），对外输出稳定错误码/可读信息。禁止吞错。
### 6.2 并发与资源治理
- 任何 goroutine 必须有生命周期控制，禁止“放飞自我”。解释并发上限策略。

---

## 7. C++（高性能/底层）规则
- 默认 RAII；裸 new/delete 仅允许出现在封装边界且有测试。  
- 多线程共享状态必须清晰同步策略。

---

## 8. MySQL / PostgreSQL 规则
- 必须参数化查询；修改/新增关键 SQL 必须提供 `EXPLAIN`。  
- 新增索引要写清：读收益、写放大、存储成本。  

---

## 9. “完成”的定义（DoD：Definition of Done）
默认 DoD：
1. 编译/类型检查通过。  
2. lint/format 通过。  
3. 测试通过或提供替代验证证据。  
4. 风险与回滚清晰。

---

## 10. 文档即代码（Docs-as-Code）
- **文档与代码同源**：所有架构、API、Runbook 必须进入代码仓库并版本化。  
- **一致性同步**：当 API、DB 结构、环境变量发生变更时，必须同步更新对应的 Markdown 文档。

---

## 11. CI/CD 一致性（Docker Compose）
- 默认采用“单仓单镜像”多阶段构建。  
- 本地开发、CI 验证、生产运行尽可能使用统一的 `docker-compose.yml` 入口。  
- 锁定依赖版本，区分环境配置，不将敏感信息硬编码进镜像。

---

## 12. UI/设计体系（Tailwind + shadcn/ui）
- **Token 化**：禁止在业务代码里大量使用随意的任意值，优先使用 CSS 变量与 Tailwind Theme 映射。  
- **双主题支持**：所有组件必须在 Light/Dark 下均可用，主题切换策略统一。  
- **可访问性 A11y**：满足键盘交互、可见焦点、语义化 ARIA。

---

## 13. 个人知识库与第二大脑约束 (Obsidian LLM Wiki)
当在处理或维护用户的 Obsidian 第二大脑时，必须强制遵循基于 `claude-obsidian` 和 Karpathy LLM Wiki Pattern 的工作流：

### 13.1 核心机制
1. **Ingest（知识摄入）**：当用户提供原始资料（如网页、PDF、文档）时：
   - 提取实体 (Entities) 和概念 (Concepts)，分别在 `wiki/entities/` 和 `wiki/concepts/` 下创建或更新卡片。
   - 在 `wiki/sources/` 下创建原始资料的元数据卡片。
   - 必须通过双向链接 (`[[链接]]`) 将知识网络缝合。
2. **Query（带源检索）**：回答用户提问时：
   - 必须先读取 `wiki/hot.md` 和 `wiki/index.md` 获取上下文。
   - 所有的合成回答必须穿透并引用具体的 Wiki 页面，严禁大模型凭空捏造。
3. **Lint（大脑体检）**：定期检查孤立页面、死链以及冲突知识，并在日志中报告。

### 13.2 页面结构与元数据 (Frontmatter)
所有写入 Obsidian 的笔记必须包含标准的 YAML Frontmatter，例如：
```yaml
---
type: <source|entity|concept|domain|meta>
title: "标题"
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags:
  - "#domain/领域名"
status: <seed|developing|mature|evergreen>
related:
  - "[[相关页面]]"
---
```

### 13.3 维护原则
- **不可变原则**：严禁修改用户的原始输入文件（位于 `.raw/` 目录），只负责生成和更新 `wiki/` 目录下的提炼卡片。
- **状态同步**：每次知识摄入（Ingest）后，必须强制更新：
  - 全局索引页：`wiki/index.md` 和对应的 `wiki/domains/<domain>/_index.md`。
  - 操作日志：在 `wiki/log.md` 顶部追加操作记录。
  - 热缓存：覆盖更新 `wiki/hot.md`。
