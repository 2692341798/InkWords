# 文档即代码与注释规范

## 1. 代码自解释与注释规范
- **Go 后端**：所有公开的（Public）函数、结构体、接口必须编写标准的 `Godoc` 注释。
- **React 前端**：所有核心 Hooks、复杂组件必须包含 `JSDoc` 注释。**前端界面的所有展示文本必须使用中文。**
- **复杂业务逻辑**：必须在代码块上方编写清晰的单行/多行注释，重点解释“为什么这么做 (Why)”，而非仅仅是“做了什么 (What)”。

## 2. 文档即代码 (Documentation as Code)
- **核心基准文档**：本项目拥有六份核心基准文档：
  1. `InkWords_PRD.md` (产品需求)
  2. `InkWords_Architecture.md` (架构设计)
  3. `InkWords_Database.md` (数据库设计)
  4. `InkWords_API.md` (接口规范)
  5. `InkWords_Development_Plan_and_Log.md` (开发计划与日志)
  6. `InkWords_Conversation_Log.md` (AI 对话与决策摘要)
- **代码与文档强同步**：在后续迭代中，若发生：
  1. 数据库表结构变更。
  2. 新增核心业务场景或修改原有产品逻辑。
  3. 接口路由/参数调整。
  4. 完成一个新的开发阶段，或遇到重大架构踩坑与修复。
  - **强制要求**：AI 必须在修改代码的**同一个执行上下文中**，主动更新对应的 Markdown 文档，并在 `Log` 日志文件中记录开发进展与决策，确保“文档即代码”的一致性，防止后续接手的开发者或 AI 产生上下文幻觉。
