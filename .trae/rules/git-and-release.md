# Git 提交与发布规范

## 1. 提交前检查与对比
- **强制对比**：在执行 `git commit` 之前，必须使用 `git diff` 或相关工具，明确本次修改与上一次提交（或暂存区）的具体区别，梳理出变更的核心逻辑和影响范围。

## 2. Commit Message 编写规范
- **遵循 Angular 规范**：严格遵循 **Conventional Commits** 规范，使用 `feat:`, `fix:`, `docs:`, `refactor:`, `chore:` 等标准前缀。
- **详实描述**：Commit Message 必须包含详细的上下文。首行简明扼要概括变更；空一行后，详细描述“Why（为什么修改）”和“What（具体修改了什么逻辑或模块）”，必要时提供中英文结合的说明。
- **原子化提交**：确保每次提交是一个原子化的变更（即只包含一个独立的功能或修复），禁止将多个不相关的修改混杂在同一个 Commit 中。

## 3. 标签 (Tag) 与推送流程
- **查询现有标签**：在推送代码或发布新版本前，必须使用 `git tag` 查询当前项目已有的标签列表，了解版本号的迭代进度（如 `v1.0.0`, `v1.0.1`）。
- **打标签与推送**：根据当前功能的量级（遵循 Semantic Versioning 语义化版本控制，Major.Minor.Patch），为项目打上合适的新标签。
- **完整推送**：最后，将代码与标签一并推送到远程仓库（使用 `git push` 和 `git push --tags`），确保代码流与版本标记的完全同步。
