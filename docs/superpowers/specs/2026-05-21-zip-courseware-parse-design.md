# InkWords ZIP 课件解析设计

## 1. 背景与目标
- 用户上传的课件通常是系列文件（多个文档与代码文件），而非单一 PDF。
- 当前系统仅支持单文件上传解析，无法直接处理 ZIP 课件包。
- 目标是在不破坏现有 `parse -> analyze -> generate` 主链路的前提下，新增 ZIP 解析能力。

## 2. 范围与非范围
### 2.1 本次范围（做）
- 支持上传 `.zip` 课件包并自动解析。
- 解析对象包含：
  - 文档类：`.pdf`、`.docx`、`.md`、`.markdown`、`.txt`
  - 代码/文本类：`.go`、`.js`、`.ts`、`.tsx`、`.jsx`、`.py`、`.java`、`.cpp`、`.c`、`.h`、`.hpp`、`.rs`、`.sql`、`.sh`、`.json`、`.yaml`、`.yml`
- ZIP 内文件自动扫描、去重、按顺序聚合，最终产出统一 `source_content`。
- 解析结果继续复用现有 SSE 分析与系列生成流程。

### 2.2 非范围（不做）
- 不支持 `.doc`（老式 Word 二进制格式）。
- 不支持 `rar/7z/tar.gz` 等其他压缩格式。
- 不做图片 OCR、音视频转写。
- 不做“手动勾选文件”交互（首版为全自动筛选）。

## 3. 方案选择
- 采用 **方案 B：后端统一 ZIP 管线**。
- 前端只做最小改动（允许 ZIP 上传与提示文案调整），核心逻辑全部在后端落地。
- 理由：
  - 与当前架构兼容性最好；
  - 风险收敛在后端，前端状态复杂度最低；
  - 易于复用现有 `source_content` 驱动的下游链路。

## 4. 架构设计
### 4.1 接口层
- 复用现有接口：`POST /api/v1/project/parse`。
- 判断上传文件扩展名：
  - 非 ZIP：走现有 `DocParser.Parse`。
  - ZIP：走新增 `ZipParser.ParseArchive`。

### 4.2 解析管线（ZIP）
1. 将上传 ZIP 写入临时文件（系统临时目录）。
2. 解压到临时目录并做路径安全校验（防路径穿越）。
3. 遍历解压结果，按白名单筛选可解析文件。
4. 对单文件执行文本提取（按现有文档解析器和纯文本读取策略）。
5. 进行内容规范化与查重去重。
6. 按顺序聚合为统一 `source_content`（保留文件边界标记）。
7. 返回 `source_content + archive_summary`。
8. 删除临时 ZIP 与临时解压目录（阅后即焚）。

### 4.3 顺序规则
- 优先按 ZIP 内路径顺序处理；
- 对同级文件名采用自然排序（如 `lesson2` 在 `lesson10` 前）。

## 5. 去重与筛选策略
### 5.1 有效文件定义
- 文件后缀在白名单中；
- 文件可成功提取文本；
- 文本长度达到最小阈值；
- 内容不与已保留文件重复或高度近似。

### 5.2 内容规范化
- 统一换行符为 `\n`；
- 去除首尾空白；
- 压缩连续空行；
- 统一后再计算指纹，避免格式差异影响重复判断。

### 5.3 去重层级
- 第一层：**完全重复去重**
  - 基于规范化文本指纹，完全一致则只保留第一份。
- 第二层：**近似重复过滤（保守）**
  - 对满足长度阈值的文本计算近似相似度摘要；
  - 高相似时视为冗余副本并丢弃。
- 策略原则：宁可少删，不误删有效补充内容。

## 6. 返回结构设计
- 保持兼容已有字段：`source_content`。
- 新增解析摘要字段：`archive_summary`，建议结构如下：

```json
{
  "source_content": "...",
  "archive_summary": {
    "total_files": 0,
    "supported_files": 0,
    "kept_files": 0,
    "duplicate_files": 0,
    "ignored_files": 0,
    "failed_files": 0,
    "kept_paths": []
  }
}
```

## 7. 前端交互设计
- 上传组件 `accept` 新增 `.zip`。
- 文案更新为中文：支持单文件或 ZIP 课件包。
- 上传成功后沿用现有流程，不新增复杂页面。
- 可选增强：在状态区域显示 `archive_summary` 的中文摘要。

## 8. 错误处理与容错
- 单文件解析失败不应导致整包失败，记录到 `failed_files`。
- 仅在“无任何有效文本文件”时返回整体失败。
- 错误提示要求用户可读、可操作，例如：
  - `压缩包中没有可解析的文本文件`
  - `压缩包内文件均为空、重复或不受支持`

## 9. 安全与资源限制
- 限制上传 ZIP 大小（沿用现有上传限制并补充后端校验）。
- 限制解压后总大小、文件数量、单文件最大解析体积。
- 拒绝路径穿越条目（如 `../`、绝对路径）。
- 防 zip bomb：对解压膨胀比和总量设置上限。
- 全流程临时文件清理，确保不持久化原始课件实体。

## 10. 测试与验收
### 10.1 单元测试
- ZIP 混合多类型文件可正确筛选与聚合。
- 完全重复文件去重正确。
- 近似重复过滤命中但不过度误删。
- 含不支持类型时可忽略且不崩溃。
- 恶意路径条目被拦截。

### 10.2 集成验证
- 上传 ZIP -> 返回 `source_content` 与 `archive_summary`。
- 继续调用 `/api/v1/stream/analyze` 能正常生成大纲。
- 大文件与高文件数场景下系统稳定，不出现 OOM 或超时失控。

### 10.3 验收标准
- 用户可直接上传系列课件 ZIP 并进入后续分析。
- 重复内容显著减少，解析摘要可反映过滤效果。
- 前后端现有单文件链路行为不回归。

## 11. 风险与回滚
- 风险：
  - 近似去重阈值过高可能误删“相似但有增量”的文件。
  - ZIP 安全防护不完整会引入资源与路径风险。
- 回滚：
  - 保留原单文件解析路径不变；
  - 可通过开关先关闭 ZIP 分支，仅保留单文件能力。
