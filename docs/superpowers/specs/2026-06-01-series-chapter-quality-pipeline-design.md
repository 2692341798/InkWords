# 系列博客单篇质量提升流水线设计

## 1. 背景

当前 InkWords 的系列博客生成能力已经具备：
- 长材料分析与大纲生成
- 按章节生成系列子文章
- 流式输出与自动续写
- 基础的润色与复习能力

但对于“系列中的单篇子文章”，当前仍存在两个核心质量短板：
- 深度不够：能讲清主题，但经常没有讲透“为什么这样设计”“关键机制如何运作”“有哪些取舍和边界”
- 案例太弱：缺少能支撑观点的代码、命令、输入输出、失败场景与复现步骤

用户的目标不是单纯提高字数，而是提升系列中每一篇子文章的知识密度、案例强度和可复现性，同时保持系列内的风格一致与成本可控。

## 2. 目标

### 2.1 核心目标
- 提升系列博客中每一篇子文章的技术深度。
- 提升系列博客中每一篇子文章的案例质量与复现价值。
- 让系列内不同章节的质量更稳定，而不是出现明显波动。
- 在质量提升的同时，利用 DeepSeek 的 KV Cache 机制降低重复上下文带来的成本与时延。

### 2.2 成功标准
- 单篇子文章在“深度”和“案例完整性”上具备稳定的最低门槛。
- 每篇子文章都能清楚回答其在整个系列中的角色与边界。
- 系列生成链路不退化为“全文反复重写”的高成本方案。
- 同一系列内的请求前缀高度稳定，缓存命中率明显提升。

## 3. 范围

### 3.1 In Scope
- 系列博客单篇子文章的生成前、生成中、生成后质量提升链路设计
- 章节理解、章节写作、章节审稿、章节统稿四阶段职责划分
- 结构化输入输出格式设计
- 质量门禁设计
- `deepseek-v4-flash` 与 `deepseek-v4-pro` 的分工策略
- KV Cache 友好的前缀复用与批次化策略

### 3.2 Out of Scope
- 重新设计整个系列博客产品 UI
- 一次性引入复杂多 Agent 编排平台
- 对非系列单篇生成链路做完整重构
- 将审稿结果直接暴露为必须人工处理的复杂工作台

## 4. 方案选择

选择方案：`前中后四段式质量流水线`

流水线定义：
1. 章节理解
2. 章节写作
3. 章节审稿
4. 定向补强与轻统稿

为什么选择该方案：
- 它能直接对应当前真实问题：深度不够与案例太弱
- 它允许在前、中、后三个环节分别施加约束，而不是把所有责任都压给一次生成
- 它比“全文多次重生成”更可控，能把高成本模型集中用在最值钱的阶段
- 它天然适合引入系列级共享上下文，提高缓存命中率

## 5. 总体架构

系列不再只是一个大纲容器，而是每篇子文章共享的上下文母体。每次章节请求都由两层输入组成：
- 系列级输入：系列总目标、目标读者、系列总大纲、统一术语、统一写作规范、统一质量门禁
- 章节级输入：当前章节标题、本章相关材料、本章与前后章节的边界、本阶段任务指令

整体链路如下：

1. 先对当前章节做结构化理解，明确“这章为什么存在”
2. 基于理解结果和章节材料生成初稿
3. 对初稿做技术深度审稿与案例完整性审稿
4. 按审稿意见做定向补强，并进行轻量统稿，产出终稿

该架构的关键原则：
- 每一阶段只承担一个明确职责
- 审稿必须基于章节理解结果回看，而不是泛泛点评
- 终稿阶段只做定向增强与统一，不做整篇重写

## 6. 阶段职责

### 6.1 阶段一：章节理解
- 输入：
  - 系列总目标
  - 目标读者画像
  - 系列总大纲
  - 当前章节标题
  - 当前章节相关源码/文档切片
  - 前后章节摘要
- 输出：
  - 本章核心目标
  - 读者真正想搞懂的问题
  - 必须解释透的机制
  - 必须出现的案例
  - 需要避免重复的内容
  - 与前后章节的衔接方式
- 责任：
  - 先决定“本章该讲什么、不该讲什么”
  - 为后续写作与审稿提供章节级约束基线

### 6.2 阶段二：章节写作
- 输入：
  - 章节理解结果
  - 当前章节原始材料
  - 系列级统一写作要求
- 输出：
  - 章节初稿
  - 覆盖范围自检
  - 示例清单
- 责任：
  - 生成可读、可发布的章节初稿
  - 强制把“原理解释、设计取舍、示例、复现、边界”写入骨架

### 6.3 阶段三：章节审稿
- 输入：
  - 章节初稿
  - 章节理解结果
  - 系列级统一质量标准
- 输出：
  - 技术深度问题列表
  - 案例问题列表
  - 结构与重复问题列表
  - 必须修复动作列表
- 责任：
  - 识别“看起来像文章，但并没有讲透”的地方
  - 识别“提到了案例，但案例并不支撑核心观点”的地方

### 6.4 阶段四：定向补强与轻统稿
- 输入：
  - 章节初稿
  - 审稿意见
  - 系列级风格与术语约束
- 输出：
  - 终稿
  - 已修复问题记录
  - 残余风险记录
- 责任：
  - 只修补审稿指出的关键缺口
  - 统一本篇标题层级、术语表达、段落节奏与系列衔接

## 7. 输入输出格式

### 7.1 章节理解输出格式

建议输出为结构化 JSON 对象，至少包含：

```json
{
  "chapter_goal": "这一章真正要解决的问题",
  "reader_questions": ["读者最关心的问题 1", "问题 2"],
  "must_explain": ["必须讲透的概念或机制"],
  "must_include_examples": ["必须出现的案例、命令、代码或图"],
  "avoid_overlap": ["不应与前后章节重复展开的内容"],
  "bridge_context": {
    "from_previous": "与上一章的衔接点",
    "to_next": "给下一章铺垫的内容"
  }
}
```

### 7.2 章节写作输出格式

建议输出为“正文 + 自检元数据”的组合对象：

```json
{
  "draft_markdown": "Markdown 正文",
  "coverage_check": {
    "goal_covered": true,
    "mechanism_explained": true,
    "examples_present": true,
    "repro_present": true,
    "edge_cases_present": true
  },
  "example_inventory": [
    {
      "example_type": "code",
      "supports_claim": "支撑了哪个关键观点"
    }
  ]
}
```

### 7.3 章节审稿输出格式

```json
{
  "depth_issues": ["深度问题 1"],
  "example_issues": ["案例问题 1"],
  "structure_issues": ["结构问题 1"],
  "revision_actions": ["必须补的动作 1"],
  "scorecard": {
    "depth": 4,
    "examples": 3,
    "reproducibility": 4,
    "clarity": 4
  }
}
```

### 7.4 补强与统稿输出格式

```json
{
  "final_markdown": "终稿 Markdown",
  "resolved_issues": ["已修复的问题"],
  "residual_risks": ["仍需人工关注的问题"]
}
```

## 8. 质量门禁

### 8.1 阶段门禁
- 章节理解阶段若没有产出 `must_explain` 或 `must_include_examples`，不允许进入写作阶段
- 章节写作阶段若没有具体案例，不允许进入终稿
- 审稿阶段若发现高优先级深度问题或案例问题，不允许跳过补强直接出终稿

### 8.2 全局硬门禁
- 深度门禁：每篇至少讲清一个“机制为什么这样设计”
- 案例门禁：每篇至少有一个能支撑核心观点的具体例子
- 复现门禁：每篇至少提供一种验证方式、运行结果或输入输出
- 边界门禁：每篇至少提一个常见坑、限制或失败场景
- 系列门禁：每篇都要能回答“它在整个系列里承担什么角色”

### 8.3 审稿要求
- 审稿意见必须指向具体缺口，不允许泛泛评价“可以更深入”
- 审稿意见必须能转化为明确的 `revision_actions`
- 审稿不得以全文重写替代问题定位

## 9. 缓存命中率设计

提升缓存命中率被视为该方案的横向优化原则，而不是附属优化。

### 9.1 固定系列级前缀
以下内容应尽量固定、复用并前置：
- 系列总目标
- 目标读者画像
- 系列总大纲
- 统一术语表
- 统一写作规范
- 统一质量门禁

### 9.2 章节级变量后置
以下内容作为后置变量：
- 当前章节标题
- 当前章节材料切片
- 当前阶段任务说明
- 当前审稿意见

### 9.3 模板标准化
- 不同阶段应复用统一的系列级 Prompt 框架
- 不要在不同章节之间随意重写同一组规则
- 不要频繁调整固定前缀中的顺序与措辞

### 9.4 批次化处理
- 同一系列的章节理解与初稿生成应尽量集中处理
- 同一系列的审稿与补强应尽量在缓存有效期内完成
- 避免长时间间隔后再零散补生成，降低缓存复用机会

### 9.5 观测指标
- 记录 `prompt_cache_hit_tokens`
- 记录 `prompt_cache_miss_tokens`
- 对同一系列不同章节的缓存命中率做对比

## 10. 模型分工与成本控制

### 10.1 `deepseek-v4-flash`
适合承担：
- 章节理解
- 章节初稿
- 轻量结构检查
- 初步案例补全

原因：
- 请求量大
- 并发需求高
- 成本更低
- 更适合铺量阶段

### 10.2 `deepseek-v4-pro`
适合承担：
- 技术深度审稿
- 高价值章节的定向补强
- 终稿轻统稿

原因：
- 真正决定成稿质量的环节更值得花高质量模型预算
- 审稿与补强是提升“深度不够、案例太弱”的关键价值环节

### 10.3 成本原则
- 不把所有阶段都放到 `pro`
- 不让补强阶段演化成全文重写
- 高并发跑草稿，中低并发跑质量关卡
- 把高成本投入集中在深度判断与案例补足上

## 11. 时延与并发策略

### 11.1 推荐执行顺序
- `flash` 串联完成章节理解与章节初稿
- `pro` 接手关键质量关卡
- 补强仅针对问题点处理，不重新生成整篇

### 11.2 并发策略
- 同一系列的多个章节可以并发跑阶段一和阶段二
- 阶段三与阶段四应降低并发，避免高质量模型被同时打满

### 11.3 为什么不会过重
- 每篇不是“重复重写三遍”
- 而是“理解一次、写一次、审一次、定向补一次”
- 增加的是判断与修复质量，而不是重复生成总量

## 12. 实施优先级

### 12.1 第一阶段
- 引入章节理解
- 引入章节写作门禁
- 引入结构化审稿输出

### 12.2 第二阶段
- 引入案例定向补强
- 引入系列级缓存友好前缀模板

### 12.3 第三阶段
- 引入轻统稿
- 引入缓存命中率观测与质量评分面板

## 13. 风险与缓解

### 13.1 风险：链路变长，时延上升
- 缓解：
  - 将 `pro` 聚焦到少数高价值步骤
  - 限制补强为定向修改

### 13.2 风险：审稿过于宽泛，无法落地
- 缓解：
  - 强制输出结构化问题列表
  - 强制产出 `revision_actions`

### 13.3 风险：缓存命中率不稳定
- 缓解：
  - 固定系列级前缀
  - 保持章节请求模板稳定
  - 同系列尽量批量处理

### 13.4 风险：统稿阶段冲掉案例与深度
- 缓解：
  - 统稿仅允许轻量统一，不允许全文重写
  - 统稿前保留审稿问题与修复清单

## 14. 验证方式

实现后至少验证以下指标：
- 审稿阶段能稳定识别深度问题与案例缺口
- 终稿中“机制解释”“具体案例”“复现方式”“边界情况”覆盖率提升
- 系列内不同章节的质量评分波动缩小
- 同一系列下的缓存命中率高于无模板复用时的基线
- 单篇平均时延虽增加，但处于可接受范围

## 15. 结论

本方案通过“章节理解 -> 章节写作 -> 章节审稿 -> 定向补强与轻统稿”的四段式流水线，直接对准系列单篇子文章的核心问题：深度不够、案例太弱。

它的价值不在于单纯拉长字数，而在于：
- 把高价值上下文提前组织好
- 把质量要求显式结构化
- 把审稿与补强变成可执行的阶段
- 用缓存友好的前缀复用机制降低重复上下文的成本

这使得 InkWords 能在不演化为复杂多 Agent 平台的前提下，显著提升系列博客中每一篇子文章的成稿质量与一致性。
